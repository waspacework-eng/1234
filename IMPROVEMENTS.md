## Детальное описание улучшений

### IMP-01 — Исправление деплоя на Railway (BUG-01)

**Проблема:** при первом деплое бот не запускался — два последовательных падения.

**Падение 1 — `ModuleNotFoundError`:**
```
ModuleNotFoundError: No module named 'aiogram'
File "/app/bot.py", line 13, in <module>
    from aiogram import Bot, Dispatcher, F
```
В репозитории отсутствовал `requirements.txt` — Railway не знал какие библиотеки устанавливать. Бот падал в цикле перезапусков.

**Решение:** добавлен `requirements.txt`:
```
aiogram>=3.0
openai>=1.0.0
python-dotenv>=1.0.0
```

**Падение 2 — Railway убивал процесс через 30 секунд:**
После добавления зависимостей Railway запускал бота как `web`-сервис и ждал открытый HTTP-порт. Бот работает через long polling — порт не открывает. Railway интерпретировал это как сбой и убивал процесс.

**Решение:** добавлен `Procfile` с явным указанием типа процесса:
```
worker: python bot.py
```
Тип `worker` сообщает Railway что процесс не обслуживает HTTP-запросы.


### IMP-02 — Исправление формата ответа ИИ (BUG-02)

**Проблема:** GPT-4o-mini игнорировал инструкцию возвращать JSON и отвечал произвольным текстом:
```
Вот несколько событий для вас!
1. Нетворкинг-завтрак — 1 апреля в 9:00...
2. Митап Python — 3 апреля в 19:00...
Ссылки: https://t.me/...
```
`json.loads()` падал с исключением -> пользователь видел сообщение об ошибке вместо карточек.

**Причина:** промпт не содержал явного запрета на текст вне JSON, модель добавляла вступительные фразы и оборачивала JSON в ` ```json ``` `.

**Решение:**

1. Переработан промпт — добавлены строгие инструкции:
```
Верни ТОЛЬКО валидный JSON и ничего кроме него.
Никакого текста до или после. Никаких ```json``` обёрток.
```

2. Добавлена функция `strip_json()` для очистки на случай если модель всё равно добавит обёртку:
```python
def strip_json(raw: str) -> str:
    raw = raw.strip()
    if "```" in raw:
        parts = raw.split("```")
        raw = parts[1] if len(parts) > 1 else parts[0]
        if raw.startswith("json"):
            raw = raw[4:]
    return raw.strip()
```

3. Добавлен `try/except` с fallback на plain-text форматирование — если JSON все равно сломан, пользователь видит читаемый список событий вместо ошибки.

### IMP-03 — Исправление галлюцинаций ИИ (BUG-03, W-04)

**Проблема:** ИИ получал все 350 событий и сам решал что подходит. Зафиксированные примеры несоответствий:

| Запрос | Что выдал ИИ |
|--------|-------------|
| «мастер-класс по лепке из глины» | IT-хакатон, нетворкинг-завтрак |
| «хочу на гончарный мастер-класс» | Speed networking, лекция «Как работает интернет» |
| «что-нибудь творческое» | Инвестиционный форум |

**Причина:** при большом объеме данных в контексте GPT-4o-mini путался в приоритетах и выбирал события по случайным признакам.

**Решение — двухэтапный поиск:**

```
Было:
  Запрос -> ИИ смотрит все 350 событий -> выбирает (галлюцинирует)

Стало:
  Запрос
    -> Шаг 1: ИИ парсит критерии — теги, ключевые слова, дата, бюджет
    -> Шаг 2: Python фильтрует базу по системе очков
    -> Шаг 3: ИИ получает 3–5 отфильтрованных событий и пишет текст
```

**Система очков релевантности (Шаг 2):**

| Совпадение | Очки |
|-----------|------|
| Тег запрошен и есть у события | +10 |
| Ключевое слово в названии | +15 |
| Ключевое слово в описании | +5 |
| Тег запрошен, но у события его нет | исключается полностью |

**Честный ответ при отсутствии результатов (W-04):**
Если ничего не нашлось — бот не придумывает замену, а явно сообщает об этом и показывает ближайшее похожее событие в другой период:
```
😔 На запрашиваемый период таких событий нет.
Ближайшее похожее — Мастер-класс: гончарное колесо с нуля (2026-04-07).
Попробуй расширить период или выбери другую тему.
```

### IMP-04 — Фильтрация прошедших событий (BUG-04)

**Проблема (от пользователей U2, U6):** бот показывал события прошедших дней и события сегодняшнего дня, время которых уже прошло. Пример: запрос в 23:00 -> событие сегодня в 10:00 попадало в выдачу.

**Решение:** добавлена функция `is_event_upcoming(event)`:

```python
def is_event_upcoming(event: dict) -> bool:
    now = datetime.now()
    event_dt = datetime.strptime(
        f"{event['date']} {event.get('time', '23:59')}", "%Y-%m-%d %H:%M"
    )
    return event_dt > now
```

Функция применяется во всех трех режимах: кнопочный поиск, `/ask`, `/surprise`. Прошедшие события не попадают в выдачу ни при каком сценарии.


### IMP-05 — Память контекста диалога (BUG-05)

**Проблема (от пользователей U7, U4):** при уточнении запроса («а подешевле?», «покажи что-нибудь похожее») бот начинал поиск заново, не понимая что уточняется предыдущий результат.

**Решение:** хранение истории сообщений в `user_data[uid]["history"]`. При каждом запросе в GPT передаются последние 8 сообщений диалога:

```python
messages=[
    {"role": "system", "content": system_prompt},
    *u["history"][-8:],  # контекст разговора
]
```

Ассистент фиксирует в истории какие события уже показал — при уточнении «покажи еще» они не повторяются. Добавлена кнопка «Сбросить историю» для явного начала нового поиска.


### IMP-06 — Защита FSM от неожиданных сообщений (BUG-06)

**Проблема:** если пользователь писал текст в момент выбора тегов или периода — бот молчал, состояние зависало и выйти из него можно было только через `/start`.

**Решение:** добавлены хэндлеры для каждого FSM-состояния и общий fallback:

```python
@dp.message(Form.choosing_tags)
async def unexpected_in_tags(message: Message):
    await message.answer("👆 Выбери теги из кнопок выше или нажми «Готово».")

@dp.message(Form.choosing_date)
async def unexpected_in_date(message: Message):
    await message.answer("👆 Выбери период из кнопок выше.")

@dp.message()
async def fallback_handler(message: Message, state: FSMContext):
    if await state.get_state() is None:
        await message.answer(
            "Не понял тебя 🤔\nИспользуй /start или /ask для поиска.",
            reply_markup=main_menu_keyboard().as_markup()
        )
```
